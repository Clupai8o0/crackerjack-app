# Conventions

Code standards, file structure, git workflow, and the small decisions that compound. When something here conflicts with `CLAUDE.md`, `CLAUDE.md` wins.

---

## TypeScript

- Strict mode, always (`"strict": true` in `tsconfig.json`)
- No `any`, no `@ts-ignore` without a comment explaining why
- Prefer `type` over `interface` unless declaration merging is genuinely needed
- Discriminated unions over boolean flags: `{ status: 'idle' } | { status: 'loading' } | ...` over `{ isLoading, isIdle, ... }`
- `as const` on literal objects you're going to consume types from
- Named exports for utilities, default export for components
- Prefer `function` declarations for top-level functions, arrow functions for callbacks

```ts
// good
export function formatPrice(amount: number, currency = 'INR'): string { ... }

// bad
export const formatPrice = (amount: number, currency = 'INR') => { ... }
```

### Where does new code go?

- **Used in one screen, never reused** → inline in the screen file or a sibling
- **Used by one feature, multiple screens within that feature** → `features/<feature>/components/`
- **Used by multiple features** → `components/patterns/`
- **No semantic content, pure UI primitive** → `components/ui/`
- **Hook called by multiple features** → `hooks/`
- **Hook called by one feature** → `features/<feature>/hooks/` (create the dir if needed)

When in doubt, put it close to where it's used. Move outward only on the second use.

## Naming

- **Components**: PascalCase, file matches name. `BookingCard.tsx` exports `BookingCard`.
- **Hooks**: camelCase starting with `use`. `useBookingStatus`.
- **Utilities**: camelCase. `formatPrice`, `parseRazorpayWebhook`.
- **Constants**: SCREAMING_SNAKE for true constants, camelCase for config-like.
- **Types**: PascalCase. Prefix with `T` only for generics (`TItem`).
- **Files for non-components**: kebab-case. `format-price.ts`, `parse-razorpay-webhook.ts`.
- **Routes (in `app/`)**: kebab-case. `sign-in.tsx`, `forgot-password.tsx`.
- **DB tables**: snake_case plural. `booking_events`, not `bookingEvents` or `booking_event`.
- **DB columns**: snake_case. Match the data model doc exactly.
- **Edge Functions**: snake_case. `create_order`, `razorpay_webhook`.
- **Migrations**: `NNNN_short_description.sql`. Four-digit prefix, snake_case rest.

## Imports

- Absolute imports via path aliases (`@/components`, `@/features`, `@/lib`, `@/hooks`, `@/types`)
- Relative imports only within the same feature module or component pair
- Order (Biome enforces):
  1. External packages
  2. Internal absolute (`@/...`)
  3. Internal relative (`./...`)
  4. Type-only imports (separate group, `import type { ... }`)

```ts
import { useState } from 'react';
import { View } from 'react-native';
import { useQuery } from '@tanstack/react-query';

import { Button, Text } from '@/components/ui';
import { supabase } from '@/lib/supabase';

import { BookingCard } from './BookingCard';

import type { Booking } from '@/types/database';
```

## State management decision tree

For any piece of state, walk the tree:

1. **Came from Supabase?** → TanStack Query. Always.
2. **Form input?** → react-hook-form. Always.
3. **Ephemeral, single component?** → `useState`.
4. **Cross-component within a feature?** → Zustand store in `features/<feature>/store.ts`.
5. **Cross-feature?** → Reconsider. There's almost always a way to lift it to a query or pass via route params.

Forbidden:
- React Context for state (use Zustand)
- Redux (we don't have it; don't add it)
- `useReducer` for anything more complex than a toggle (use Zustand)

## Data fetching

All Supabase reads go through TanStack Query. All writes go through mutations. No exceptions.

```ts
// features/artist/queries.ts
export function useArtistProfile(id: string) {
  return useQuery({
    queryKey: ['artist', id],
    queryFn: async () => {
      const { data, error } = await supabase
        .from('artist_profiles')
        .select('*, profile:profiles(*), categories:artist_categories(category)')
        .eq('profile_id', id)
        .single();
      if (error) throw error;
      return data;
    },
  });
}
```

Query keys are stable, shallow arrays. First element is the entity, then qualifiers. `['artist', id]`, `['bookings', 'organizer', userId, status]`.

Invalidation:
- After a mutation, invalidate the most specific key that was affected. Don't `invalidateQueries()` with no key.
- For optimistic updates: only on writes the server is unlikely to reject (e.g. read receipts, profile edits). Never on bookings or payments.

## Forms

react-hook-form + zod. Schema lives in `features/<feature>/schema.ts`.

```ts
// features/auth/schema.ts
export const signInSchema = z.object({
  email: z.string().email('Enter a valid email'),
  password: z.string().min(8, 'At least 8 characters'),
});
export type SignInValues = z.infer<typeof signInSchema>;
```

```tsx
// app/(auth)/sign-in.tsx
const { control, handleSubmit, formState } = useForm<SignInValues>({
  resolver: zodResolver(signInSchema),
});
```

Always use `<FormField>` (the design system primitive) — never raw `<Controller>` + custom layout. The primitive handles label / error / disabled states uniformly.

## Error handling

Three categories:

1. **Expected errors** (validation failures, "wrong password") — surface to user via the form or a non-blocking inline message. Don't log to Sentry.
2. **Operational errors** (network down, Supabase 500) — surface via toast-equivalent (the design system has no toast — instead, use an inline error state on the screen) and log to Sentry.
3. **Programmer errors** (a `null` you didn't expect, an unreachable case) — let the error bubble; Sentry catches it via the global handler. Add a Sentry breadcrumb just before, so you have context.

```ts
// good
try {
  await supabase.functions.invoke('create_order', { body: payload });
} catch (err) {
  Sentry.captureException(err, { tags: { feature: 'booking' } });
  setFormError('Could not start payment. Try again.');
}
```

Custom errors live in `lib/errors.ts`:
```ts
export class PaymentError extends Error { ... }
export class NotAuthorizedError extends Error { ... }
```

## Git workflow

- **Branch from `main`.** Branch names: `<type>/<short-description>`. Examples: `feat/sign-in-screen`, `fix/portfolio-reorder`, `db/0007-payments-index`.
- **Trunk-based.** Branches live ≤ 3 days. If a branch is older, rebase onto `main` or merge it.
- **Squash on merge.** Single commit per PR on `main`.
- **Commit messages**: Conventional Commits with phase tag.
  ```
  feat(P1): sign-in screen with email + password
  fix(P2): portfolio reorder no longer loses order on app restart
  db(P1): add 0001_init.sql with all V1 tables
  chore: bump expo SDK to 52
  docs: update CLAUDE.md commands section
  ```
- **PR titles** match the squash commit message.
- **PR descriptions** include:
  - What it does (1-2 lines)
  - How to test it (the owner reads this)
  - Screenshots if UI
  - Any decisions made that aren't covered in docs (and a note to add them to `decisions.md` if structural)

## Testing

V1 testing strategy is intentionally light:

- **No unit tests for UI** until a regression bites twice
- **Unit tests for `lib/format.ts`-style pure functions** — yes, with vitest
- **Unit tests for Edge Functions** — yes, especially payment / webhook logic. In `supabase/functions/_tests/`
- **Integration tests for RLS** — yes. SQL test file run against local Supabase, asserting that anon/artist/organizer roles can/can't read each table
- **E2E (Maestro/Detox)** — defer until post-MVP

Run tests:
```bash
pnpm test                       # all unit tests
pnpm test:functions             # edge function tests
pnpm test:rls                   # RLS integration tests
```

CI runs all three on every PR.

## Analytics events

Naming: `<entity>_<action>` snake_case. Past tense.

Examples:
- `app_opened`
- `signup_completed`
- `role_selected`
- `artist_profile_completed`
- `portfolio_item_uploaded`
- `search_performed`
- `booking_request_submitted`
- `booking_accepted`
- `payment_initiated`
- `payment_succeeded`
- `payment_failed`
- `payout_released`
- `review_submitted`

Properties: include relevant context but never PII. `{ artist_id, category, city, price_band }` is fine. `{ artist_email, artist_name }` is not.

When you add a new event, add it to `lib/analytics.ts` as a typed wrapper:
```ts
export const track = {
  signupCompleted: (props: { method: 'email' | 'phone' | 'google' }) =>
    posthog.capture('signup_completed', props),
  // ...
};
```

## Comments

- **Why, not what.** Code says what. Comment when the why isn't obvious.
- **TODO without an owner is forbidden.** `// TODO(claude): handle ...` or `// TODO(P4): wire to Razorpay`.
- **No commented-out code in commits.** Delete it; git remembers.
- **JSDoc only on public-ish utilities** — exported from `lib/` or `hooks/`. Internal components don't need it.

## Accessibility minimums

Per Design System §2.3 and `05_design_notes.md`:

- WCAG AA contrast on all text (Tailwind tokens already comply; don't fight them)
- Min 44pt tap targets (the design system uses 36-pt pills with extra hit-slop; `<Pressable hitSlop={...}>`)
- Dynamic Type support — never hardcode `fontSize` outside `Text` primitive variants
- VoiceOver / TalkBack labels on every interactive element
- No reliance on color alone for state (use copy + weight, per Design System §9)

## Performance defaults

- `<FlashList>` for any list > 10 items. Never `<FlatList>`.
- `<Image>` from `expo-image`, never `react-native`'s built-in. Always specify `width` / `height` to prevent layout thrash.
- `useMemo` / `useCallback` only when there's a measured rerender problem. Don't preoptimize.
- Avoid inline functions in deeply-nested children; hoist when the component is hot.
- Hermes engine on, always.
